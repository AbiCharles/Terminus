I've got a DuckDB database of experimental sensor runs at /app/measurements.duckdb and I want a command-line tool to extract signals from it, built at /app/signal_analysis.py. The measurement data lives only in that database — there's no CSV or raw dump anywhere, so everything has to go through SQL queries.

The database has an `experiments` table (experiment_id, sample_rate_hz, n_samples, status, description) and a `measurements` table (experiment_id, sample_index, signal). Some experiments are flagged with a status other than 'valid' (things like quarantined or failed runs) and those must be skipped everywhere — only valid experiments ever get analyzed.

For this first step, make `python /app/signal_analysis.py query` pull each valid experiment's samples and write them to /app/artifacts/<experiment_id>/measurements.csv with columns sample_index,signal ordered by sample_index, alongside a /app/artifacts/<experiment_id>/metadata.json holding {"experiment_id": <string>, "sample_rate_hz": <float>, "n_samples": <int>}. The `query` subcommand takes `--db` for the database path (default `/app/measurements.duckdb`) and `--output-dir` for the output directory (default `/app/artifacts`); both are options on the subcommand, e.g. `python /app/signal_analysis.py query --db /app/measurements.duckdb --output-dir /app/artifacts`. With no `--experiment-id` it processes every valid experiment; it also takes optional repeatable `--experiment-id <id>` flags that scope processing to specific experiments, silently skipping any id that doesn't exist or isn't valid.
