Finally, add a `track` subcommand so `python /app/signal_analysis.py track` records each valid experiment's analysis as its own run in MLflow. Use a local sqlite tracking store at sqlite:////app/mlflow.db with artifacts under /app/mlruns, and put the runs under an experiment named signal-analysis. The `track` subcommand takes the same `--db` (default `/app/measurements.duckdb`) and `--output-dir` (default `/app/artifacts`) options plus repeatable `--experiment-id`, e.g. `python /app/signal_analysis.py track --db /app/measurements.duckdb --output-dir /app/artifacts`; the data still comes only from /app/measurements.duckdb.

Name each run after its experiment id and log the analysis you computed in the previous step. Log params for experiment_id, sample_rate_hz, n_samples, detrend, the bootstrap seed (mc_seed), iteration count (mc_iterations), and confidence level (confidence_level). Then score each run for signal quality and promote only the clean ones. Define a run's spectral SNR as the fraction of the detrended signal's FFT power (excluding the DC bin) carried by its three strongest frequency components, and log it as a metric named spectral_snr. Apply an acceptance gate: an experiment is accepted when its spectral_snr is at least 0.80 and rejected otherwise. Tag each run with status set to accepted or rejected accordingly.

For each accepted experiment, fit a Fourier reconstruction model and register it. The model is an ordinary least-squares linear regression whose target is the detrended signal and whose features are, for each of the three dominant frequencies in descending-magnitude order, the pair [cos(2*pi*f*t), sin(2*pi*f*t)] where t is each sample's time in seconds (sample_index / sample_rate_hz) — six features in the order [cos f1, sin f1, cos f2, sin f2, cos f3, sin f3].
