Milestone 2 — train and log the candidate roster.

Using the policy you wrote to `/app/policy.json` and the dataset at `/app/data.csv`, train every candidate in the roster and log each one as its own run in the MLflow tracking store at `sqlite:////app/mlflow.db`, under the experiment named in the policy. Partition the data with the dataset's `split` column — fit on the `train` rows and compute all acceptance metrics and fairness statistics on the `test` rows. Use the dossier's preprocessing exactly: standardize the numeric features with `StandardScaler` and one-hot encode the categoricals with `OneHotEncoder(handle_unknown="ignore")`, default settings otherwise.

For each candidate run, log: the candidate's roster name (its identifier in the policy's candidate roster, e.g. `logreg_baseline`, `gbm_audit`, `gbm_compliant`) as a `model_type` param — this is the candidate identity, not the scikit-learn estimator class; the `estimator` name (the scikit-learn class, e.g. `LogisticRegression`) as its own separate param; and each of the estimator's hyperparameters as its own flat, top-level MLflow param under its exact scikit-learn name (for example `random_state`, `C`, `max_iter`, `n_estimators`, `max_depth`, `learning_rate`) — do not prefix or nest them. Log the `accuracy`, `f1_macro`, and `roc_auc` metrics, and one demographic-parity-difference metric per protected attribute named `bias_dpd_<attribute>`, where the protected attributes are exactly the `column` values listed under the policy's `bias_slices` (e.g. `bias_dpd_region`). Also log a `validation_report.json` artifact for the run and the fitted model itself under the artifact path the policy specifies (`credit_default_model`).

Train each candidate on the policy's approved feature set only — the numeric and categorical predictors listed under `feature_policy`. The protected attributes (the `bias_slices` columns: `age_group`, `gender`, `region`) must NOT be used as model inputs by any candidate; they are read only to compute the fairness metrics, never trained on. The one exception is the single prohibited proxy feature (`region_risk_index`): it must be used as a model input only by the one candidate the dossier flags as the proxy-using audit model, and by no other candidate.
